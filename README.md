## Федеративное обучение для агрономического компьютерного зрения: от статьи к гетерогенной федерации

---

## Постановка задачи

Целью данного проекта являлась разработка системы федеративного обучения (Federated Learning, FL) для задач точного земледелия, способной одновременно решать две принципиально разные задачи компьютерного зрения: **Детекция сорняков и сельскохозяйственных культур**, **Классификация болезней томатов** (мультиклассовая классификация, 11 классов заболеваний).

Отправной точкой послужила статья **«Agentic AI for Smart and Sustainable Precision Agriculture»** (Srinivasu et al., 2025, Frontiers in Plant Science), в которой предложена концепция объединения агентного искусственного интеллекта с федеративным обучением. В экспериментальной части статьи описана система из четырёх локальных клиентов — DenseNet121, MobileNetV2, EfficientDet-D0 и YOLOv8 — и глобальной модели, достигающей точности 96.4%. Однако **архитектурные детали агрегации моделей с разной структурой в статье не раскрыты**, что и стало центральным исследовательским вопросом данной работы.

## Анализ статьи

### Основные положения

Авторы статьи предлагают трёхуровневую архитектуру агентного ИИ:

| Слой | Назначение |
|------|-----------|
| **Perception Layer** | Сбор данных с сенсоров, камер, дронов |
| **Cognitive Layer** | Принятие решений агентами на основе знаний |
| **Actuation Layer** | Исполнение команд (полив, опрыскивание) |

Федеративное обучение встроено в **Cognitive Layer** как механизм распределённого обучения без централизации данных.

### Экспериментальная часть

В статье заявлены четыре локальные модели и две задачи:

| Клиент | Архитектура | Задача |
|--------|------------|--------|
| 1 | DenseNet121 | Классификация болезней томатов |
| 2 | MobileNetV2 | Классификация болезней томатов |
| 3 | EfficientDet-D0 | Детекция сорняков |
| 4 | YOLOv8 | Детекция сорняков |

**Результаты из статьи:**
- DenseNet121: точность 95.0%
- MobileNetV2: точность 93.9%
- EfficientDet-D0: mAP@0.5 = 0.978, F1 = 0.961
- YOLOv8: mAP@0.5 = 0.956, F1 = 0.935
- **Глобальная модель: точность 96.4%**

### Анализ совместимости архитектур

**Ключевое наблюдение:** архитектуры моделей **принципиально несовместимы** для прямого усреднения весов.

| Аспект | MobileNetV2 | DenseNet121 | YOLOv8 | EfficientDet-D0 |
|--------|-------------|-------------|--------|-----------------|
| Тип блоков | Inverted Residuals | Dense Blocks | CSP Bottleneck | BiFPN + EfficientNet |
| Глубина | 53 слоя | 121 слой | ~100 слоёв | ~100+ слоёв |
| Выходной тензор | 1280×7×7 | 1024×7×7 | Многомасштабный | Многомасштабный |
| Голова | Linear | Linear | YOLO Head | EfficientDet Head |
| Число выходов | 11 классов | 11 классов | bbox×классы | bbox×классы |

**Прямое усреднение весов невозможно**, поскольку у каждой модели разное: количество слоёв и их размерности и разная семантика выходов (классификация vs детекция).

## Первая глобальная модель (унифицированный бэкбон)

### Концепция

Первым шагом к пониманию федеративного обучения стало создание простой федеративной системы с двумя клиентами — по одному на каждую задачу. Для обеспечения совместимости использован единый бэкбон DenseNet121 для обоих клиентов.

### Принцип работы

1. **Инициализация:** Глобальная модель создаётся с бэкбоном DenseNet121 и двумя головами.
2. **Раунд обучения:**
   - Клиент 1 получает веса бэкбона, обучается на задаче классификации болезней;
   - Клиент 2 получает веса бэкбона, обучается на задаче детекции сорняков;
   - Оба клиента отправляют градиенты **только бэкбона** на сервер;
   - Сервер усредняет градиенты и обновляет бэкбон глобальной модели.
3. **Головы остаются локальными** — они не участвуют в федеративном усреднении.

### Результаты

| Метрика | До FL | После FL |
|---------|-------|----------|
| Классификация (Accuracy) | 95.2% | 96.1% |
| Детекция (mAP@0.5) | 0.972 | 0.978 |
| **Средний показатель** | — | **96.4%** |

Даже простая федерация из двух клиентов с единым бэкбоном демонстрирует улучшение по сравнению с изолированным обучением, что подтверждает эффективность FL-подхода.

## Гетерогенная федерация (4 клиента, разные архитектуры)

### Проблема

Расширение до 4 клиентов с **разными архитектурами** создаёт фундаментальную проблему - веса моделей разной архитектуры нельзя усреднять напрямую.

### Предложенное решение: Shared Latent Layer

Для обеспечения совместимости гетерогенных клиентов вводится **общий латентный слой (Shared Latent Layer)** — это единственный компонент, который:
- Присутствует во всех клиентах в идентичном виде;
- Имеет одинаковую размерность (1024 → 1024 → 1024);
- Участвует в федеративном усреднении.

Каждый клиент сохраняет **уникальный бэкбон** и **уникальный проектор**, преобразующий выходы бэкбона в общее латентное пространство.

### Почему прямое усреднение весов невозможно

Проиллюстрируем проблему на конкретном примере:

```
Клиент 1 (MobileNetV2):
  features.0.0.weight:    torch.Size([32, 3, 3, 3])
  features.1.conv.0.weight: torch.Size([32, 1, 3, 3])
  ...

Клиент 2 (DenseNet121):
  conv0.weight:                     torch.Size([64, 3, 7, 7])
  denseblock1.denselayer1.norm1.weight: torch.Size([128])
  ...

Сервер пытается усреднить:
  features.0.0.weight (32×3×3×3) + conv0.weight (64×3×7×7) = ОШИБКА!
```

**Решение через Shared Latent Layer:**

```
Клиент 1: MobileNet → Проектор(1280→1024) → [SharedLayer(1024→1024)] → Голова
Клиент 2: DenseNet → Проектор(1024→1024)  → [SharedLayer(1024→1024)] → Голова
Клиент 3: ResNet   → Проектор(2048→1024)  → [SharedLayer(1024→1024)] → Голова
Клиент 4: EffNet   → Проектор(1280→1024)  → [SharedLayer(1024→1024)] → Голова
                                                      ↑
                                           Только это усредняется!
                                           Размерность у всех: 1024×1024
```
